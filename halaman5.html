01

Pemrograman Berorientasi Obyek

Definisi OOP

Paradigma pemrograman yang meniru realita di dunia nyata — di mana kehidupan dipandang sebagai objek yang beraktifitas secara mandiri maupun saling berhubungan.

📐
Structural
Class
Template / Master / Blueprint dari sebuah objek. Mendefinisikan struktur yang akan dimiliki semua objek dari class tersebut.

Terdiri dari dua komponen utama:
  • Atribut — data / ciri khusus objek
  • Method — perilaku / aksi objek
🧍
Instance
Object
Perwujudan nyata dari sebuah class. Satu class dapat digunakan untuk membuat banyak objek.

Setiap objek yang dibuat dari class yang sama akan memiliki struktur atribut dan method yang identik, namun nilai atributnya berbeda.

Contoh — Class Mahasiswa (Java)

class Mahasiswa { // === ATRIBUT === String nama; String nim; String jurusan; Float ipk; // === METHOD === void info() { System.out.println( "Nama: " + nama); System.out.println( "NIM: " + nim); } void belajar() { System.out.println(nama + " sedang belajar."); } }
// Membuat objek dari class Mahasiswa Mahasiswa andri = new Mahasiswa(); Mahasiswa budiman = new Mahasiswa(); andri.nama = "Andri"; andri.nim = "1123121"; andri.jurusan = "Sistem Informasi"; andri.ipk = 3.75f; budiman.nama = "Budiman"; budiman.nim = "33345454"; budiman.info(); // memanggil method budiman.belajar();
01.2

Super Class & Sub Class (Inheritance)

Inheritance adalah konsep "mewariskan" atribut dan method dari Super Class ke Sub Class. Sub Class akan memiliki semua atribut/method induk, ditambah miliknya sendiri.
MANUSIA (SuperClass)
nama : String
umur : int
berjalan()
makan()
→
DOSEN (SubClass)
nip : String
mataKuliah : String
mengajar()
memberiNilai()
MAHASISWA (SubClass)
nim : String
jurusan : String
ipk : float
belajar()
lihatIPK()
class Dosen extends Manusia { // "extends" = mewarisi dari Manusia String nip; String mataKuliah; void mengajar() { System.out.println(nama + " sedang mengajar " + mataKuliah); } }
01.3

Method Overriding

Method Overriding adalah mendefinisikan ulang method yang diwarisi dari superclass dengan perilaku baru yang lebih spesifik di subclass. Nama method, parameter, dan return type harus sama.
// Di Superclass: class Manusia { void makan() { System.out.println("Manusia makan untuk bertahan hidup."); } } // Di Subclass — override method makan(): class Dosen extends Manusia { @Override void makan() { System.out.println("Dosen makan sambil membaca jurnal penelitian."); } }
Syarat Overriding
1. Nama method harus sama
2. Parameter harus sama
3. Return type harus sama
4. Gunakan @Override annotation
Hasilnya
Manusia.makan() →
"Manusia makan..."

Dosen.makan() →
"Dosen makan sambil..."
02

UML — Unified Modeling Language

Definisi UML

Bahasa standar untuk memvisualisasikan, menspesifikasikan struktur & perilaku, serta mendokumentasikan perancangan perangkat lunak berbasis objek.

Alasan #1
Perencanaan
Perencanaan adalah 50% dari keberhasilan. UML membantu merencanakan sistem sebelum coding dimulai.
Alasan #2
Menyamakan Persepsi
Menyamakan persepsi antar developer yang mungkin berbeda pemahaman tentang sistem yang akan dibangun.
Alasan #3
Panduan Teknis
Menjadi panduan dalam membangun sistem — Class Diagram → Class di Java, juga bisa dikonversi ke ERD Database.

Jenis Diagram dalam UML

Static
Structural Diagram
Fokus pada apa yang ada di sistem. Bersifat statis — tidak berubah seiring waktu.

Class Diagram Object Diagram Component Diagram Deployment Diagram Package Diagram
Dynamic
Behaviour Diagram
Fokus pada apa yang terjadi di sistem. Bersifat dinamis — menggambarkan alur dan logika proses.

Use Case Diagram Activity Diagram Sequence Diagram State Diagram Communication Diagram
03

Use Case Diagram

Diagram perilaku yang memodelkan fungsionalitas / fitur yang disediakan sistem, serta hubungan antara fungsionalitas tersebut dengan aktor yang berinteraksi. Digunakan pada tahap analisis sistem.
Komponen 1
👤 Aktor (Actor)
Entitas eksternal (manusia atau sistem lain) yang berinteraksi dengan sistem. Digambarkan sebagai stick figure.
Komponen 2
⭕ Use Case
Fungsionalitas atau layanan yang disediakan sistem untuk aktor. Digambarkan sebagai elips dengan nama di dalamnya.
Komponen 3
⬜ Sistem (Boundary)
Kotak persegi yang memisahkan bagian internal sistem dari aktor eksternal. Use case berada di dalam kotak.
Komponen 4
↔ Relasi
Koneksi antara aktor dan use case, atau antar use case. Ada 4 jenis: Association, Include, Extend, Generalization.

Jenis-jenis Relasi

Relasi Digunakan Antara Simbol Fungsi Utama
Association Aktor ↔ Use Case Garis lurus Menunjukkan interaksi langsung antara aktor dan use case
Include Use Case → Use Case «include» Use case selalu menyertakan use case lain. Memisahkan proses yang umum/berulang.
Extend Use Case → Use Case «extend» Use case dapat memperluas use case lain secara opsional (kondisional)
Generalization Aktor ↔ Aktor / UC ↔ UC Segitiga putih Hubungan pewarisan — anak mewarisi hubungan dan perilaku dari induk
04

Class Diagram

Diagram struktural yang memodelkan struktur statis sistem — menunjukkan kelas, atribut, operasi, dan hubungan antar kelas. Setelah Use Case Diagram selesai, Class Diagram digunakan untuk merancang bagaimana sistem dibangun secara teknis.

Anatomi Class Box

NamaKelas
- namaAtribut : tipeData
# protectedAtrib : String
+ publicAtrib : int
+ namaMethod() : void
- privateMethod() : String
# protectedMethod() : int

Visibility (Aksesibilitas)

Simbol Nama Akses
+ Public Dari mana saja / luar kelas
- Private Hanya dari dalam kelas
# Protected Kelas itu sendiri & turunannya

Hubungan Antar Class

Relasi Simbol UML Makna
Asosiasi Garis lurus Hubungan logis antar kelas (bisa dua arah atau satu arah)
Generalisasi Segitiga ke atas Pewarisan (Inheritance) — kelas anak mewarisi dari kelas induk
Agregasi Belah ketupat putih "Memiliki" — bagian dari, tapi bagian bisa berdiri sendiri
Komposisi Belah ketupat hitam Bagian tidak bisa berdiri sendiri — hilang jika induk hilang
05

Activity Diagram

Diagram perilaku yang memodelkan alur kerja / proses aktivitas dalam sistem. Menggambarkan urutan aktivitas, pengambilan keputusan, percabangan, dan penggabungan proses. Sering disebut flowchart versi UML.

Komponen Utama

Komponen Simbol / Bentuk Fungsi
Initial Node ● Lingkaran hitam kecil Titik awal dari proses / aktivitas
Activity ⬭ Persegi panjang membulat Representasi satu langkah aktivitas
Decision Node ◇ Belah ketupat Titik pengambilan keputusan (percabangan dua arah atau lebih)
Merge Node ◇ Belah ketupat Penggabungan alur sebelum melanjutkan ke aktivitas berikutnya
Fork Node ━ Garis tebal Memecah alur menjadi beberapa aktivitas paralel
Join Node ━ Garis tebal Menyatukan beberapa aktivitas paralel menjadi satu alur
Final Node ⊙ Lingkaran berlapis Menandai akhir dari seluruh proses
Control Flow → Panah Menunjukkan arah / urutan alur proses
06

Hubungan Antar Diagram

Kunci Penting

Ketiga diagram saling melengkapi dan terhubung secara logis — masing-masing memberikan sudut pandang berbeda terhadap sistem yang sama.

Step 1 — Analisis
Use Case Diagram
Menjawab: "Apa yang harus dilakukan sistem?"

Mengidentifikasi aktor, fitur/fungsi, dan ruang lingkup sistem dari sudut pandang pengguna.
Step 2 — Desain
Class Diagram
Menjawab: "Bagaimana sistem dibangun?"

Memodelkan kelas, atribut, method, dan relasi antar komponen sistem. Dikonversi jadi Class Java & ERD Database.
Step 3 — Detail Proses
Activity Diagram
Menjawab: "Bagaimana alur prosesnya?"

Menjabarkan langkah-langkah di dalam setiap use case atau method, agar urutan implementasi lebih konkrit.

Contoh — Sistem Peminjaman Buku Online

Elemen UML Contoh Diagram
Use Case "Pinjam Buku" Use Case Diagram
Activity Login → Cari Buku → Cek Ketersediaan → Pinjam Activity Diagram
Kelas Terkait User, Buku, TransaksiPeminjaman Class Diagram
Ingat: Seorang developer tidak mungkin bisa membuat Class tanpa membuat Use Case terlebih dahulu. Untuk menterjemahkan method/fungsi juga diperlukan Activity Diagram agar urutan langkahnya lebih konkrit.
